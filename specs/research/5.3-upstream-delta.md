# Lua 5.3 as a Delta from Lua 5.4

Research spec for porting Lua 5.3 as a configurable variant of the existing 5.4 Rust
implementation. The authoritative spine is the Lua 5.4 reference manual §8
"Incompatibilities with the Previous Version" (which enumerates the 5.3 → 5.4 changes),
cross-checked against the upstream C source for both versions.

## 0. Executive framing

5.3 and 5.4 are **very close** languages — far closer than 5.1↔5.3. They share the dual
int/float number model, integer division `//`, bitwise operators, `goto`, `utf8`,
`string.pack`/`unpack`/`packsize`, and `__idiv`/`__band`/`__bor`/`__bxor`/`__shl`/`__shr`/`__bnot`
metamethods. The genuine divergences cluster in five areas:

1. **Bytecode/opcode set** — totally different (47 vs 83 opcodes, different encoding). This is the
   biggest single difference but it is **invisible to a from-scratch VM** unless we expose
   `string.dump`/`load` of precompiled chunks or assert on disassembly. We are past the
   C-mirroring phase, so our internal bytecode need not match either version's; only the
   *observable* `string.dump` byte format is a structural-oracle concern.
2. **`math.random`** — different RNG algorithm and seeding (the single most impactful runtime
   behavioral delta; output streams will never match between versions).
3. **Numeric `for` over integers** — wraparound semantics changed.
4. **Stdlib surface** — `warn`, `coroutine.close`, `bit32`, `__ipairs`, `collectgarbage`
   options, generational GC, `math.*` compat funcs, `io.lines` arity, `print` internals.
5. **A handful of language/metamethod nuances** — string→number coercion location,
   decimal-literal overflow, `__lt`-emulates-`__le` removal, goto-into-tbc-scope.

For a "5.3 backend behind the unified embedding API," the practical work is: gate the stdlib
surface, swap the RNG, restore the old `for`-loop semantics, restore the old coercion/literal
rules, and reinstate `bit32`. The lexer needs essentially no change; the parser needs only the
removal of `<const>`/`<close>` attribute parsing.

## 1. Number model

Both 5.3 and 5.4 use the dual-subtype model: `number` is either a 64-bit `integer` or a
64-bit `float` (IEEE double), with `LUA_32BITS` available in both. The shared rules:

- Integer arithmetic **wraps around** (two's complement) on overflow in both.
- Bitwise ops convert operands to integers (error if a float has no exact integer
  representation), operate on all 64 bits, and yield an integer — **identical in both**.
- `//` (floor division) and `%` (modulo with sign of divisor) — identical.
- Division `/` and exponentiation `^` always produce floats in both.
- A float with an exact integral value used as a **table key is normalized to the integer
  key**: `a[2.0]` stores under integer `2`. Verified in both (`ltable.c` `luaH_finishset` /
  `luaV_tointeger` path; NaN key raises "table index is NaN" in both). `3` and `3.0` are the
  same key in both versions.
- `string.format`/`tostring` format floats with `LUA_NUMBER_FMT` = `"%.14g"` in **both**
  (verified: `luaconf.h` in both trees). A float that prints without a `.`/`e` gets a `.0`
  appended (`lua_number2str`/`tostringbuff` logic) in both — `tostring(3.0)` → `"3.0"`,
  `tostring(3)` → `"3"`, in both versions. **No formatting delta here.**

### Real divergences in the number model

**1a. String → number coercion location (5.4 §8.1):**
> "The coercion of strings to numbers in arithmetic and bitwise operations has been removed
> from the core language. The string library does a similar job for arithmetic operations
> using metamethods. However, unlike in previous versions, the new implementation preserves
> the implicit type of the numeral in the string. For instance, the result of `"1" + "2"`
> now is an integer, not a float."

In **5.3**, the VM itself coerces strings in arithmetic (`luaO_arith`/`luaV_tonumber_`),
and the result type follows the normal numeric-promotion rules. In **5.4**, the core no
longer coerces; the `string` library installs arithmetic metamethods on the string type. For
a 5.3 backend, coercion must live in the core arithmetic path and `"1" + "2"` must yield an
**integer** in 5.3 too (5.3 already preserves implicit type — this bullet is mostly about
*where* the coercion happens, and that bitwise on strings works in 5.3). Net observable
difference is small but exists for edge cases (e.g. coercion in bitwise: `"3" & 5` works in
5.3 core, in 5.4 only via the string metamethod which it also supplies).

**1b. Decimal integer literal overflow (5.4 §8.1):**
> "The use of the `__lt` metamethod to emulate `__le` has been removed. … literal decimal
> integer constants that overflow are read as floats, instead of wrapping around."

In **5.3**, a decimal integer literal that overflows `LUA_MAXINTEGER` **wraps around** to an
integer. In **5.4**, it is read as a **float**. (Hex literals wrap in both.) So in 5.3,
`9223372036854775808` (= maxint+1) reads as a negative integer (wrap); in 5.4 it reads as the
float `9.2233720368548e+18`. The 5.3 backend must wrap decimal literal overflow in the lexer.
Verified location: `llex.c` `read_numeral` → `luaO_str2num`.

**1c. Numeric `for` over integers (5.4 §8.1):**
> "The semantics of the numerical for loop over integers changed in some details. In
> particular, the control variable never wraps around."

This is the headline. **5.3** (`lvm.c` `OP_FORLOOP`): the integer index is updated with
`intop(+, idx, step)` which **wraps**, then compared `idx <= limit`. A loop like
`for i = math.maxinteger-1, math.maxinteger do ... end` works, but
`for i = 1, math.maxinteger, big_step` can wrap the index past the limit and **loop forever
or terminate wrongly**. **5.4** precomputes an unsigned iteration count before the loop
(`forprep`) using `(limit - init) / step` as an unsigned division, so the control variable
**never wraps** and a zero step is an error. The 5.3 backend must restore the wrapping
add-and-compare behavior (and 5.3 does **not** error on step==0 the way 5.4's prep does —
5.3 with step 0 raises "'for' step is zero" at FORPREP? **Verify**: 5.3 raises
`'for' step is zero` only if step is integer 0 in `OP_FORPREP`? Actually 5.3 checks step != 0
in `forprep`/`luaG`; confirm exact message — see Open Question 3).

Verified 5.3 source: `lua-5.3.6/src/lvm.c` lines ~1190–1245 (`OP_FORLOOP` uses
`intop(+, ivalue(ra), step)`; `forlimit()` clamps a float limit to MAXINTEGER/MININTEGER).

**1d. `tostring` of integers vs floats:** identical (`"3"` vs `"3.0"`), see above.

## 2. Lexer / tokens

**Essentially no difference.** Both have the same reserved-word set:
`and break do else elseif end false for function goto if in local nil not or repeat return
then true until while`. `goto` is a reserved word in **both** (verified `llex.c` token list in
5.3.6). The `<const>`/`<close>` *attributes* added in 5.4 are **NOT** new tokens — `const`
and `close` are ordinary identifiers matched as strings inside `<...>` by the parser
(`lparser.c` `getlocalattribute` does `strcmp(attr, "const")`). The `<` and `>` tokens
already exist in both for comparison operators.

Numeric/string literal lexing is the same (hex floats, `\u{XXXX}`, long brackets, `\z`).
**Action: the 5.3 lexer is the 5.4 lexer unchanged.**

## 3. Parser / syntax

| Construct | 5.3 | 5.4 |
|---|---|---|
| `local x <const> = e` | **syntax error** | supported |
| `local x <close> = e` | **syntax error** | supported (to-be-closed) |
| `goto` / labels `::name::` | supported | supported |
| Integer division `//`, bitwise `& | ~ << >>` | supported | supported |
| Everything else | identical grammar | identical grammar |

The **only** parser delta is the local-variable attribute syntax `<attrib>` introduced in
5.4 (`lparser.c` `getlocalattribute`, `RDKCONST`/`RDKTOCLOSE`). For a 5.3 backend, the parser
must **reject** `<const>`/`<close>` as a syntax error (in 5.3, `local x <` parses `<` as an
unexpected token → "'=' or 'in' expected" / "unexpected symbol near '<'").

**Goto scope rule (5.4 §8.1, a parser-level check):**
> "A label for a goto cannot be declared where a label with the same name is visible, even if
> this other label is declared in an enclosing block."

Also 5.4 added: a `goto` cannot jump into the scope of a to-be-closed (or any) local — but
since 5.3 has no `<close>`, the relevant 5.3 rule is just the classic "cannot jump into the
scope of a local." The stricter same-name-label-in-enclosing-block rule is 5.4-only; 5.3 is
more permissive here. There is **5.3-only** lenience, not 5.3-only syntax.

There is **no 5.3-only syntax** that 5.4 lacks (5.4 is a strict superset grammatically, modulo
the attribute syntax).

## 4. Bytecode / opcode set

This is the largest raw difference, but it is **internal** and only matters at the
`string.dump`/`load`-precompiled boundary (structural oracle), not for behavior.

| Aspect | 5.3 | 5.4 |
|---|---|---|
| Opcode count (`NUM_OPCODES`) | **47** | **83** |
| Instruction word | 32-bit, fields `OP`(6) `A`(8) `B`(9) `C`(9) / `Bx`(18) / `sBx` / `Ax`(26) | 32-bit, fields `OP`(7) `A`(8) `B`(8) `C`(8) `k`(1) / `Bx`(17) / `sBx` / `sJ`(25) / `Ax` |
| Constant access | **RK(B)/RK(C)** — high bit of a B/C operand selects "constant vs register" | **no RK bit**; instead dedicated `*K` opcodes (`ADDK`, `GETFIELD`, `SETFIELD`, `GETTABUP`, `SELF`, `EQK`, …) |
| Arithmetic | one `OP_ADD`/`SUB`/… each, operands RK | split into register form (`ADD`), const form (`ADDK`), immediate form (`ADDI`/`SHRI`/`SHLI`), plus the `MMBIN`/`MMBINI`/`MMBINK` follow-up |
| Metamethod dispatch | inline in each arith op via `luaT_trybinTM` | **`OP_MMBIN`/`OP_MMBINI`/`OP_MMBINK`** — the arith op runs the fast path then, if it needs a metamethod, a *following* `MMBIN*` instruction performs the call (new in 5.4) |
| Booleans | `OP_LOADBOOL A B C` (with optional skip) | `OP_LOADFALSE`, `OP_LFALSESKIP`, `OP_LOADTRUE` (3 ops) |
| Small int / float literals | via constant table (`LOADK`) | `OP_LOADI` (immediate sBx int), `OP_LOADF` (immediate float) |
| Comparisons | `OP_EQ/LT/LE A B C` (RK operands, A = expected bool) | `OP_EQ/LT/LE A B k` + `OP_EQK`, `OP_EQI`, `OP_LTI/LEI/GTI/GEI` (immediate comparisons) |
| `for` loop | `OP_FORPREP`/`OP_FORLOOP` (`A sBx`) — wrapping integer index | `OP_FORPREP`/`OP_FORLOOP` (`A Bx`) — precomputed unsigned count |
| Generic `for` | `OP_TFORCALL` + `OP_TFORLOOP` | `OP_TFORPREP` + `OP_TFORCALL` + `OP_TFORLOOP` (adds TFORPREP that creates an upvalue for the closing variable) |
| Return | single `OP_RETURN A B` | `OP_RETURN`(`A B C k`), `OP_RETURN0`, `OP_RETURN1` (fast paths) |
| Upvalue close | encoded in `OP_JMP A sBx` (if A>0, close upvalues ≥ R(A-1)) | dedicated `OP_CLOSE A`; plus `OP_TBC A` (mark to-be-closed) — 5.4-only |
| Vararg | `OP_VARARG A B` | `OP_VARARG A C` + `OP_VARARGPREP A` (adjusts vararg params at function entry) — VARARGPREP is 5.4-only |
| Table set immediate | `OP_SETLIST A B C` with `LFIELDS_PER_FLUSH` and FPF index math | `OP_NEWTABLE A B C k`, `OP_SETLIST A B C k`, `OP_GETI`/`OP_SETI` (integer-key fast path), `OP_SETFIELD`/`OP_GETFIELD` (short-string-key) — all new fast paths |
| Extra arg | `OP_EXTRAARG Ax` | `OP_EXTRAARG Ax` (same idea) |

**Dump/undump header:** both write a header via `ldump.c`/`lundump.c` with a version byte
(`0x53` vs `0x54`), `LUAC_DATA` "\x19\x93\r\n\x1a\n", and size/format bytes. **The dumped
bytecode is not cross-loadable between versions** (different version byte and instruction
layout). 5.3 `lua_Integer`/`lua_Number` test values in the header are the same constants
(`0x5678`, `370.5`).

**Porting implication:** our VM is already a from-scratch design, not a byte-for-byte mirror
of either C VM. So opcodes are an *internal* choice. The only place the opcode set leaks into
observable behavior is:
- `string.dump(f)` / `load(dumped)` round-trips and any cross-process precompiled chunk.
- The official `code.lua` test (asserts on internal opcode selection / `getobjname`
  disassembly) — but that test only runs under the internal-tests (`ltests`/`testC`) build
  (`code.lua` line 5: "testC not active: skipping opcode tests"), so it is **not** part of
  the behavioral oracle for a normal binary. **We can ignore opcode-level parity.**

## 5. Standard library surface diff

Verified against `linit.c`, `lbaselib.c`, `lmathlib.c`, `lcorolib.c`, `lstrlib.c`,
`ltablib.c`, `loslib.c`, `liolib.c`, `lutf8lib.c` in both trees.

### Libraries opened by default (`linit.c`)

| Library | 5.3 | 5.4 | Note |
|---|---|---|---|
| base, package, coroutine, table, io, os, string, math, utf8, debug | yes | yes | |
| **`bit32`** | **yes** (`LUA_BITLIBNAME "bit32"`, `luaopen_bit32`) | **removed** | 5.3-only |

### Basic library (`lbaselib.c` `base_funcs[]`)

| Function | 5.3 | 5.4 |
|---|---|---|
| `warn` | **absent** | present |
| `loadstring` | present under `LUA_COMPAT_LOADSTRING` (alias of `load`) | removed |
| `assert error getmetatable ipairs loadfile load next pairs pcall print rawequal rawlen rawget rawset select setmetatable tonumber tostring type xpcall collectgarbage dofile` | present | present |
| `_G`, `_VERSION` | present (`_VERSION` = `"Lua 5.3"`) | present (`"Lua 5.4"`) |

- **`ipairs`/`__ipairs`:** 5.3 honors the `__ipairs` metamethod when built with
  `LUA_COMPAT_IPAIRS` (default ON in 5.3, `pairsmeta(L,"__ipairs",…)`). 5.4 **removed
  `__ipairs` entirely.** For a faithful default-5.3 backend, `ipairs` should consult
  `__ipairs`. (Most code doesn't use it; low risk.)
- **`print`:** 5.4 §8.2: *"The function print does not call tostring to format its arguments;
  instead, it has this functionality hardwired."* In **5.3**, `print` calls the global
  `tostring`, so redefining `tostring` affects `print`. In **5.4** it does not. The 5.3
  backend's `print` must route through the (overridable) `tostring`.

### `collectgarbage` options (`lbaselib.c`)

| Option | 5.3 | 5.4 |
|---|---|---|
| `stop restart collect count step isrunning` | yes | yes |
| `setpause`, `setstepmul` | yes (functional) | present but **deprecated** (no-op-ish under generational) |
| `incremental` | **absent** | present (mode switch + tuning params) |
| `generational` | **absent** | present |

`collectgarbage("count")` returns **same** in both: KB as a float (`Kbytes + bytes/1024`),
verified identical in both `luaB_collectgarbage`. `collectgarbage("step")` returns a boolean
in both. **Action:** 5.3 backend rejects `"incremental"`/`"generational"` and accepts
`setpause`/`setstepmul` as real tuning.

### `coroutine` (`lcorolib.c`)

| Function | 5.3 | 5.4 |
|---|---|---|
| `create resume running status wrap yield isyieldable` | yes | yes |
| **`close`** | **absent** | present (closes a suspended/dead coroutine, runs `__close` of its tbc vars) |

`coroutine.close` is 5.4-only and is tied to to-be-closed variables. 5.3 backend omits it.

### `math` (`lmathlib.c`)

| Function | 5.3 | 5.4 |
|---|---|---|
| `abs acos asin atan ceil cos deg exp floor fmod log max min modf rad sin sqrt tan` | yes | yes |
| `tointeger type ult` | yes | yes |
| `pi huge maxinteger mininteger` | yes (`maxinteger`/`mininteger` exist in 5.3) | yes |
| `random randomseed` | yes (C-`rand()` based) | yes (xoshiro256**) |
| `atan2 cosh sinh tanh pow frexp ldexp log10` | present under `LUA_COMPAT_MATHLIB` (**default ON in 5.3** → effectively present in a stock build) | present only under `LUA_COMPAT_MATHLIB` (**default OFF in 5.4** → effectively absent) |

So the math-surface delta is mostly about **defaults**: a stock 5.3 has `atan2`/`cosh`/…;
stock 5.4 does not. There are **no** math functions that exist in 5.4 but not 5.3 (the set
`tointeger`/`type`/`ult`/`maxinteger`/`mininteger` already landed in 5.3).

**`math.random` — the big one (5.4 §8.2):**
> "The pseudo-random number generator used by the function math.random now starts with a
> somewhat random seed. Moreover, it uses a different algorithm."

- **5.3** (`math_random`): `r = l_rand() * (1/(L_RANDMAX+1))` — a `double` in `[0,1)` from C
  `rand()`. With args, `lua_pushinteger((lua_Integer)(r * (up-low+1)) + low)`. Seeded by
  `math.randomseed(x)` → C `srand((unsigned)x)`. **No-arg `randomseed` does not exist as a
  random-seed-from-entropy** — `randomseed` requires a numeric arg in 5.3. `math.random(0)`
  → "interval is empty" **error** (since `low=1, up=0`).
- **5.4** (`math_random`): built-in **xoshiro256\*\*** 256-bit state. `math.randomseed()` with
  no args seeds from entropy (time + address); with args seeds deterministically. `math.random(0)`
  is a **special case** returning a random integer over the **entire** integer range
  (verified: `if (up == 0) { return full-range integer }`). Integer interval uses an
  unbiased "project to range" (`project()`), not `(lua_Integer)(float*range)`.

**Consequence:** `math.randomseed(0); math.random()` produces **completely different streams**
in 5.3 vs 5.4. Any oracle test that asserts exact random values will differ; the official
`math.lua` random tests are *statistical* (ranges/distribution) except where they reseed and
check determinism. For the 5.3 backend we must implement the **C-rand-equivalent** algorithm
(or, more faithfully, document that exact stream parity with C requires matching the platform
`rand()` — see Open Question 1).

### `string` (`lstrlib.c`)

Functions are the **same set** in both: `byte char dump find format gmatch gsub len lower
match rep reverse sub upper pack unpack packsize`. (`pack`/`unpack`/`packsize` exist in
**both** 5.3 and 5.4 — verified `getnumlimit`/pack machinery in 5.3.6 `lstrlib.c`.)

- `string.format("%d", 3.5)` → **error in both** (`luaL_checkinteger`); the message wording
  differs slightly (see §8). `%d` on a float with integral value (`3.0`) is **accepted in
  both**.
- `string.format("%s", v)` calls `tostring`/`__tostring` in both; 5.4 added length handling
  but the surface is the same.
- Pattern matching: identical (`%g` class added back in 5.3 already; `%f` frontier in both).
- `string.dump` byte format differs (see §4) — only matters for precompiled-chunk parity.

### `table` (`ltablib.c`)

Same set in both: `insert remove concat sort pack unpack move`. (`table.move` landed in 5.3.)
No surface delta.

### `utf8` (`lutf8lib.c`)

Present in both (`utf8.char`, `utf8.charpattern`, `utf8.codes`, `utf8.codepoint`, `utf8.len`,
`utf8.offset`). **Delta (5.4 §8.2):**
> "By default, the decoding functions in the utf8 library do not accept surrogates as valid
> code points."

In **5.3** the decoders accept surrogate code points (and accept the full original UTF-8 range
up to 6 bytes / `0x7FFFFFFF` via the lax flag in places). 5.4 tightened the default to reject
surrogates and added an explicit `lax` parameter to `utf8.codepoint`/`utf8.len`/`utf8.offset`
controlling this. For a 5.3 backend, surrogates should decode by default.

### `os` (`loslib.c`)

Same surface in both (`clock date difftime execute exit getenv remove rename setlocale time
tmpname`). No documented behavioral incompatibility in §8 for `os.*`. (5.4's `os.exit`
boolean/true-default unchanged from 5.3.)

### `io` (`liolib.c`)

Same surface. **Delta (5.4 §8.2):**
> "The function io.lines now returns four values, instead of just one."

In **5.3**, `io.lines(filename, ...)` returns **one** value (the iterator). In **5.4** it
returns four (iterator + state + control + closing value), so e.g.
`for l in io.lines(name) do` still works but `io.lines` used positionally differs. The 5.3
backend's `io.lines` returns a single value.

## 6. Metamethods diff

Shared in both: `__index __newindex __call __tostring __metatable __len __pairs __gc __mode
__name __add __sub __mul __div __mod __pow __unm __idiv __band __bor __bxor __bnot __shl __shr
__concat __eq __lt __le`.

| Metamethod | 5.3 | 5.4 |
|---|---|---|
| `__idiv`, bitwise (`__band __bor __bxor __bnot __shl __shr`) | present | present (added in 5.3, unchanged) |
| **`__close`** | **absent** | present (called when a to-be-closed variable goes out of scope or on error) |
| `__ipairs` | honored by `ipairs` under default-ON compat | **removed** |
| `__lt` emulating `__le` | **5.3 uses `__lt` to derive `__le`** when `__le` absent (`a <= b` tried as `not (b < a)`) | **removed** (5.4 §8.1): each comparison uses only its own metamethod |
| `__gc` non-function value | 5.3 ignores a `__gc` that isn't a function | **5.4 calls any value** present as `__gc` (5.4 §8.1: *"Lua does not ignore __gc metamethods that are not functions. Any value will be called"*) |
| `__eq` dispatch | only when both operands are tables (or both userdata) and raw-unequal | same |

**Dispatch differences worth restoring for 5.3:**
1. **`__le` via `__lt`:** in 5.3, if `__le` is missing but `__lt` exists, `a <= b` evaluates
   `not (b < a)` using `__lt`. The 5.4 backend dropped this. The 5.3 backend must reinstate the
   `__lt`-fallback for `<=`. (This was already deprecated/`LUA_COMPAT` in 5.3 but on by default.)
2. **`__gc` non-function:** 5.3 silently ignores; 5.4 calls. Minor; restore 5.3's ignore.
3. No `__close`/to-be-closed machinery in 5.3 at all.
4. 5.4 introduced `OP_MMBIN` opcode-level metamethod dispatch, but that is an internal
   mechanism — observable metamethod *semantics* for arithmetic are the same in both (try fast
   path; on type mismatch consult metamethod of left then right operand).

## 7. GC observable diff

| Aspect | 5.3 | 5.4 |
|---|---|---|
| Collector | incremental mark-and-sweep only | incremental **+ generational** mode |
| `collectgarbage("generational")` / `("incremental")` | **error** (no such option) | switch modes (+ optional tuning args) |
| `collectgarbage("setpause"/"setstepmul")` | functional tuning | deprecated (still parsed) |
| `collectgarbage("count")` | KB float | KB float (same) |
| `collectgarbage("step", n)` | returns boolean (was a full collection cycle done) | returns boolean (same) |
| `__gc` value handling | non-function ignored | any value called |
| Finalizer errors | propagate as `LUA_ERRGCMM` (5.3 had this constant) | **never propagated**, emitted as a **warning** (5.4 §8.3: `LUA_ERRGCMM` removed) |
| C API GC opts | `LUA_GCSETPAUSE`/`LUA_GCSETSTEPMUL` | deprecated, replaced by `LUA_GCINC`/`LUA_GCGEN` |

For a 5.3 backend, the observable GC surface is: no generational mode, the older
`collectgarbage` option list, `__gc` non-function ignored, and finalizer errors that propagate
(not warn). The actual collector *implementation* need not match (the oracle only sees timing-
insensitive observable effects like finalizer order, weak-table clearing, `collectgarbage`
return values).

## 8. Error-message-format differences the official suite asserts on

`errors.lua` and others assert on exact substrings of error messages. Known wording deltas
(verify each against both binaries when wiring the oracle):

| Situation | 5.3 message (substring) | 5.4 message (substring) |
|---|---|---|
| Arithmetic on non-number | `attempt to perform arithmetic on a ... value` | same shape; 5.4 added variable-name annotations in more cases |
| `for` step zero | `'for' step is zero` | `'for' step is zero` (same) |
| `for` limit/init not number | `'for' limit must be a number` / `'for' initial value must be a number` (5.3) | 5.4: `'for' initial value must be a number` etc. — **5.4 reworded** some (e.g. the limit message). **Verify exact strings.** |
| String→number coercion failure in arithmetic | 5.3: `attempt to perform arithmetic on a string value` (coercion failed) | 5.4: arithmetic metamethod path → message references the string library metamethod |
| `string.format("%d", 3.5)` | `bad argument #2 to 'format' (number has no integer representation)` | `bad argument #2 to 'format' (number has no integer representation)` (same in both) |
| Calling a nil value | `attempt to call a nil value` (+ name) | same, with more name info in 5.4 |
| Indexing a nil value | `attempt to index a nil value` (+ name) | 5.4 improved name resolution (upvalue/field/global qualifiers) |
| `<const>` assignment | n/a (no const) | 5.4: `attempt to assign to const variable 'x'` |

**General pattern:** 5.4 invested heavily in *better variable-name annotations* in runtime
error messages (`luaG_runerror`/`varinfo`/`getobjname` improvements). The 5.3 messages are
often shorter / less annotated. Because our suite-as-oracle is **version-specific**, we run
the 5.3 official tests against the 5.3 backend and they assert 5.3 wording — so we must match
**5.3** wording for the 5.3 backend, not 5.4. This is a per-backend message table, not a
single shared one. (`errors.lua` from the 5.3.4 suite is the spec for the exact strings.)

## The ORACLE for 5.3

### Version to pin

**Pin `lua-5.3.6`** (the final 5.3.x release, 2020-09-14; security/bug fixes over 5.3.5).
Rationale: latest 5.3.x is the most-patched and what "5.3" means today.

- Source: <https://www.lua.org/ftp/lua-5.3.6.tar.gz>
- `LUA_VERSION_NUM` = 503, `_VERSION` = `"Lua 5.3"`.

### Test suite

⚠️ **There is no published `lua-5.3.6-tests.tar.gz`** (verified: returns HTTP 404). The latest
available official test tarball in the 5.3 line is **`lua-5.3.4-tests`**:

- Test suite: <https://www.lua.org/tests/lua-5.3.4-tests.tar.gz> (HTTP 200, verified)
- This is the standard PUC-Rio behavioral oracle for the 5.3 series. Use the 5.3.4 tests
  against the 5.3.6 reference binary (and against our 5.3 backend). Behavioral diffs between
  5.3.4 and 5.3.6 are bug fixes, not language changes, so the 5.3.4 suite is a valid oracle for
  a 5.3.6 binary. Record this mismatch in `source.toml`.

### Tests known to be environment / locale / build sensitive

- **`code.lua`** — asserts on internal **opcode selection and disassembly**; only runs under
  the `ltests`/`testC` instrumented build (prints "testC not active: skipping opcode tests"
  otherwise). **Irrelevant** to a from-scratch VM with a different opcode set; skip or run only
  the non-`testC` portions.
- **`api.lua`, `db.lua`** — exercise the **C API** and debug hooks heavily; need the
  `testC`/`ltests` build and a matching C-API surface. For a Rust-native embedding API these
  are mostly N/A; run the pure-Lua subsets.
- **`math.lua`** — `math.random` tests reseed and check **distribution/ranges**, plus some
  exact-value checks after `randomseed(0)`. Our RNG ≠ C `rand()`, so any exact-stream assertion
  will fail unless we bit-match the platform `rand()`. Treat random exact-value asserts as
  expected-divergence; keep the statistical/range asserts.
- **`literals.lua`, `constructs.lua`** — touch `os.setlocale` / locale-dependent number
  parsing and `string.format`; run under `LC_ALL=C` for determinism (classic locale-sensitive
  Lua tests — the decimal-point and `%a` formatting depend on locale).
- **`files.lua`, `main.lua`** — filesystem, temp-file, and process behavior; need unique temp
  names across parallel runs and a real file path (not a `-e` string). Same caveats already
  documented in this repo's CLAUDE.md debugging playbook.
- **`big.lua`, `verybig.lua`** — memory/time heavy; gate behind a "slow tier."
- **`gc.lua`** — finalizer **order** and weak-table clearing; observable but
  implementation-tolerant. `__gc`-non-function and finalizer-error-propagation behavior must
  match **5.3** (ignore / propagate), unlike 5.4.

## Top 5 open questions / risks

1. **`math.random` stream fidelity.** 5.3's `math.random` is built on C `rand()`/`srand()`,
   which is **platform-libc-defined** — even the reference 5.3.6 binary produces different
   streams on glibc vs macOS vs MSVC. There is no portable "the 5.3 random sequence." Decide:
   (a) match the reference binary's platform `rand()` exactly (brittle, platform-locked), or
   (b) ship a documented deterministic 5.3-compatible LCG and accept that exact-value oracle
   asserts in `math.lua` diverge (treat as known-expected). Recommend (b) + quarantine the
   exact-value random asserts.

2. **No `lua-5.3.6-tests` tarball.** The newest 5.3 test suite is **5.3.4**. Confirm there are
   no 5.3.5/5.3.6 behavioral changes the 5.3.4 suite would mis-assert (spot-check `bugs` page).
   Risk is low but should be recorded in `source.toml` as a deliberate version skew.

3. **Exact `for`-loop edge semantics & messages.** Verify 5.3's behavior and exact error string
   for `for i=1,10,0` (step zero) and for float-limit clamping (`forlimit` clamps to
   MAX/MININTEGER), and confirm the 5.4 reworded `'for' limit must be a number` family so the
   per-version error table is right. The wraparound restoration must be tested against
   `for i=math.maxinteger-1,math.maxinteger` and large-step cases.

4. **Coercion placement vs the embedding API.** "Strings coerce in core arithmetic" (5.3) vs
   "string library installs metamethods" (5.4) is an architectural fork. For a *shared* codebase
   backing both versions, decide whether the core arithmetic path takes a `version` flag that
   enables string coercion, or whether 5.4 keeps the metamethod approach and 5.3 flips a core
   flag. Affects where the error message originates (§8 wording differs accordingly).

5. **Scope/cost of `bit32` and default-compat math.** Stock 5.3 ships `bit32` plus the
   `atan2/cosh/…/log10` aliases by default (compat ON). These are pure additions (low risk) but
   are real surface a "faithful 5.3" must expose, and `bit32` has its own 32-bit-wraparound
   semantics distinct from 5.3's 64-bit native bitwise operators — easy to get subtly wrong
   (e.g. `bit32.band` masks to 32 bits, native `&` is 64-bit). Budget a small `bit32`
   implementation + tests (`bitwise.lua` covers native ops; `bit32` has its own historical
   tests in older suites).
