# Lua 5.1 (non-JIT) & 5.2 upstream research — toward a multi-version embedding backend

Status: research / planning. Audience: anyone implementing the 5.1 and 5.2 backends behind
the unified embedding API. The current engine targets **5.4.7 only**; this document maps the
deltas a 5.1 backend (with 5.2 as the bridge) must honor.

Framing: **5.1 is a genuine second family**, not a feature-flag of 5.4. The reason is the
number model (float-only, no integer subtype) and the globals model (function environments +
`OP_GETGLOBAL`/`setfenv`). Both are pervasive, cross-cutting, and observable. 5.2 keeps the
float-only number model but modernizes globals to `_ENV` — so **5.2 is the bridge**: porting
5.2 first lets us reuse the modern `_ENV` globals path with a float-only numeric core, then 5.1
is "5.2 minus `_ENV`, minus goto, minus the 5.2 escapes, plus legacy stdlib names."

Pins (oracle): **Lua 5.1.5** and **Lua 5.2.4** (final point releases of each line).

Primary sources (all claims below are traceable to these):
- Lua 5.1 Reference Manual — https://www.lua.org/manual/5.1/manual.html
- Lua 5.2 Reference Manual — https://www.lua.org/manual/5.2/manual.html
- Lua 5.4 Reference Manual — https://www.lua.org/manual/5.4/manual.html
- Lua 5.1.5 source — https://www.lua.org/source/5.1/ (`luaconf.h`, `lopcodes.h`, `lundump.c`)
- Lua 5.2.4 source — https://www.lua.org/source/5.2/
- 5.2 §8 "Incompatibilities with the Previous Version" — https://www.lua.org/manual/5.2/manual.html#8

## 0. Version-family map (why 5.1 is the hard one)

| Axis | 5.1 | 5.2 | 5.3 | 5.4 (current target) |
|---|---|---|---|---|
| Numbers | float-only (double) | float-only (double) | **int/float subtypes** | int/float subtypes |
| Globals | function env + `setfenv`/`getfenv` + `OP_GETGLOBAL` | **`_ENV` upvalue** | `_ENV` | `_ENV` |
| `goto`/labels | no | **yes** | yes | yes |
| `\x`,`\z`,`\ddd` escapes | no | **yes** | yes | yes (+`\u{}` in 5.3) |
| Bitwise ops | none (lib only via 3rd party) | `bit32` library | **native `& | ~ << >>`** | native operators |
| `<close>`/`<const>` attribs | no | no | no | **yes** |
| Integer division `//` | no | no | yes | yes |
| `__len` on tables | **no (userdata only)** | yes | yes | yes |
| `__pairs`/`__ipairs` | no | **yes** | `__pairs` only (ipairs gone) | neither (`__pairs` removed in 5.4) |
| `__gc` on tables | **no (userdata only)** | **yes** | yes | yes |

The two rows in bold for 5.1 — float-only numbers and env-based globals — are what make 5.1 a
separate backend rather than a config of the 5.4 core. 5.2 shares the number model with 5.1 and
the globals model with 5.4, which is exactly why it is the bridge.

## 1. Number model — 5.1/5.2 are FLOAT-ONLY

In 5.1 and 5.2 there is **one** number type. `luaconf.h` (5.1.5) defines `LUA_NUMBER` as
`double` and there is no integer subtype; `type(x)` is `"number"` for every numeric value and
`math.type` does not exist (it is 5.3+). Source: https://www.lua.org/source/5.1/luaconf.h.html ,
5.1 manual §2.2 ("*Number* represents real (double-precision floating-point) numbers").

5.4, by contrast, has two subtypes (`integer` = 64-bit, `float` = double) and `math.type`
returns `"integer"`/`"float"`/`fail`. 5.4 manual §3.4.3. This single difference fans out into
every numeric surface:

| Surface | 5.1 / 5.2 (float-only) | 5.4 (dual) | Backend action for 5.1/5.2 |
|---|---|---|---|
| Internal repr | every number is `f64` | tagged `Integer(i64)` \| `Float(f64)` | force the numeric value to a single `f64` variant; never produce `Integer` |
| `type()` | always `"number"` | always `"number"` (subtype hidden) | same string; but `math.type` must be **absent** |
| `tostring(3)` | `"3"` | `"3"` | both print `3`; but `tostring(3.0)` differs below |
| `tostring(3.0)` | `"3"` (`%.14g` of `3.0` → `3`) | `"3.0"` (float marker) | 5.1/5.2 use `%.14g` with **no** forced `.0`; 5.4 appends `.0` to integral floats |
| Default float format | `LUA_NUMBER_FMT` = `"%.14g"` | `LUAI_NUMFFORMAT` = `"%.14g"` for floats, `%d`/`%lld` for ints | match `%.14g` exactly for all numbers in 5.1/5.2 |
| `tonumber("3")` | float `3` | integer `3` | return the single float value |
| `1/2` | `0.5` | `0.5` (float `/`) | same |
| `7 % 3` | `1` (fmod-based, float) | `1` (integer) but `7.0%3.0` float | always float `%`; `__mod` uses `luai_nummod` |
| `2^2` | `4` (`pow`, float) | `4.0` (float `^`) | always float |
| `3 == 3.0` | `true` (same value) | `true` (cross-subtype eq) | trivially true (one type) |
| `math.floor(x)` | returns a **float** (e.g. `3` printed as `3`) | returns an **integer** | return float; do not coerce to int |
| `string.format("%d", 3.5)` | accepts, truncates (lenient) | **errors** ("number has no integer representation") | replicate lenient 5.1 `%d` behavior |
| `for i=1,3 do` loop var | float counter | integer counter when bounds are ints | counter is float |
| Table key `t[3]` vs `t[3.0]` | **same slot** (one number type; float key with integral value normalizes) | same slot (5.4 normalizes float-with-integral-value to int key) | 5.1 already collapses; ensure float key `2.0` hits the array/int slot |
| Large integer literal | parsed as double (precision loss > 2^53) | parsed as 64-bit integer (exact) | accept loss; do **not** add an int path |
| Hex float `0x1p4` | 5.1 has hex **integer** literals; hex floats `0x..p..` added in **5.2** | yes | 5.1 lexer: hex int only, no `p` exponent; 5.2 lexer: hex float allowed |

**Host i64 marshaling (embedding API concern).** The unified API will accept Rust `i64`/`u64`
from host code. Under a 5.1/5.2 backend there is no integer Lua value to map to: an `i64` must be
**widened to `f64`** on the way in, and any value above 2^53 silently loses precision. This is a
genuine semantic divergence the API must surface (e.g. a `lossy` flag or a documented contract),
not paper over. On the way out, a Lua number is always `f64`; converting back to host `i64`
should follow a single documented rounding/range rule (truncate-toward-zero, error on
non-integral or out-of-range) — pick one source of truth, do not fall back. Contrast: under 5.4
an `i64` round-trips exactly as a Lua integer.

## 2. Globals model — function environments (5.1) vs `_ENV` (5.2+)

### 5.1: function environments + `setfenv`/`getfenv`
5.1 manual §2.9: "All global variables live as fields in ordinary Lua tables, called
*environment tables*. ... Each function has its own reference to an environment, so that all
global variables in this function will refer to this environment table." A global reference `x`
compiles to a lookup on the function's environment, emitted as the dedicated bytecode ops
`OP_GETGLOBAL`/`OP_SETGLOBAL` (see §4). Functions to manipulate this at runtime:
- `getfenv([f])` — returns the environment of function `f` (or stack level / running function);
- `setfenv(f, table)` — sets the environment of `f`; `f` may be a function or a stack level.
- C API: `lua_getfenv`/`lua_setfenv`, pseudo-index `LUA_GLOBALSINDEX`, `LUA_ENVIRONINDEX`.

These are the mechanism behind sandboxing in 5.1 (`setfenv(0, sandbox)` / `setfenv(f, sandbox)`)
and behind `module()`/`package.seeall`.

### 5.2: `_ENV` upvalue (modern globals), float numbers retained
5.2 manual §2.2: free names resolve through an implicit upvalue named `_ENV`; `x` is sugar for
`_ENV.x`. There is one real global table reachable via the registry
(`LUA_RIDX_GLOBALS`). `OP_GETGLOBAL`/`OP_SETGLOBAL` are **gone** — globals are just `GETTABUP`/
`SETTABUP` (table access through the `_ENV` upvalue). Sandboxing is now "compile/load the chunk
with a custom `_ENV`" (the `env` argument to `load`).

5.2 §8.1/§8.2 (verbatim): "The concept of environment changed. Only Lua functions have
environments. To set the environment of a Lua function, use the variable `_ENV` or the function
`load`. C functions no longer have environments." and "Functions `setfenv` and `getfenv` were
removed, because of the changes in environments." API: "Pseudoindex `LUA_GLOBALSINDEX` was
removed. ... Pseudoindex `LUA_ENVIRONINDEX` and functions `lua_getfenv`/`lua_setfenv` were
removed."

### Migration / backend strategy
Because 5.2 already uses `_ENV` and that path is identical in shape to 5.4, **build the 5.2
globals path first by reusing the 5.4 `_ENV` machinery**; the only difference from 5.4 is the
number core. Then the 5.1 backend layers the *legacy* env model on top:
- compiler emits `OP_GETGLOBAL`/`OP_SETGLOBAL` (or lowers them to an `_ENV`-equivalent internal
  table access whose table is the per-closure environment rather than an upvalue);
- `getfenv`/`setfenv` are present and mutate a per-closure environment pointer;
- the registry-global vs per-function-env distinction must be observable (a closure can have a
  *different* environment than the global table).

| Globals feature | 5.1 | 5.2 | 5.4 |
|---|---|---|---|
| Free-name lookup | function env table | `_ENV` upvalue | `_ENV` upvalue |
| `setfenv`/`getfenv` | present | **removed** | absent |
| `_ENV` keyword | absent | present | present |
| Global bytecode | `OP_GETGLOBAL`/`OP_SETGLOBAL` | `GETTABUP`/`SETTABUP` on `_ENV` | `GETTABUP`/`SETTABUP` |
| Sandbox idiom | `setfenv(f, t)` | `load(chunk, name, mode, env)` | `load(..., env)` |
| C-func environment | yes (`LUA_ENVIRONINDEX`) | no | no |

## 3. Syntax / lexer deltas

| Feature | 5.1 | 5.2 | 5.3/5.4 | Notes |
|---|---|---|---|---|
| `goto` / `::label::` | **no** | **yes** | yes | 5.2 added `stat ::= goto Name` and `label ::= '::' Name '::'` (5.2 grammar §9) |
| `\xXX` hex escape | no | **yes** | yes | string-literal escape; 5.1 lexer rejects `\x` |
| `\z` skip-whitespace escape | no | **yes** | yes | 5.2 §3.1 |
| `\ddd` decimal escape | yes | yes | yes | present in 5.1 already |
| `\u{XXXX}` unicode escape | no | no | **5.3+** | |
| Hex float literal `0x1.8p1` | no | **yes** | yes | 5.1 has hex **integer** literals only; the `p`/`P` binary exponent is 5.2 |
| Bitwise operators `& | ~ << >>` | no | no | **5.3** | in 5.1/5.2 these tokens are not operators |
| Integer division `//` | no | no | **5.3** | `/` is always float division |
| `<const>`/`<close>` attribs `local x <const>` | no | no | **5.4** | attrib syntax `local namelist` only in 5.1/5.2 |
| `%z` pattern class (the NUL class) | yes | **deprecated** | removed | 5.2 patterns allow literal `\0`; `%z` deprecated (5.2 §8.2) |
| Long-string/level `[==[` | yes | yes | yes | present in 5.1 |
| Locale-dependent identifiers | allowed | **disallowed** | disallowed | 5.2 §8.1: "Lua identifiers cannot use locale-dependent letters." |

**Lexer/number-literal implication for 5.1:** the numeric scanner must accept decimal floats,
decimal ints (stored as double), and **hex integers** (`0x1F`), but must **reject** the `p`
binary exponent (5.2-only) and any `0b`/`//`/bitwise tokens. All numeric tokens lower to a single
`f64` value (§1).

## 4. Bytecode / opcode format

**5.1 precompiled-chunk header** (12 bytes; `lundump.c`, 5.1.5):
`1B 4C 75 61` (`ESC "Lua"` signature) · `51` (version) · `00` (format) · endianness byte
(`01` little / `00` big) · `sizeof(int)` (4) · `sizeof(size_t)` (4 or 8) · `sizeof(Instruction)`
(4) · `sizeof(lua_Number)` (8) · integral flag (`00` = float numbers). Source:
https://www.lua.org/source/5.1/lundump.c.html . There is no `LUAC_TAIL`/data-fingerprint block
(that is 5.3+). 5.2 changes the header to add a 6-byte `LUAC_TAIL` (`19 93 0D 0A 1A 0A`) data
check and bumps the version byte to `52`.

**5.1 instruction format** (`lopcodes.h`, 5.1.5): fixed 32-bit instruction, 6-bit opcode in the
low bits, then `A` (8 bits), and either `B`/`C` (9 bits each) or `Bx` (18) / `sBx` (signed Bx).
**38 opcodes**, `OP_MOVE`(0) … `OP_VARARG`(37). The 5.1-specific ones the backend must model:

```
OP_GETGLOBAL  A Bx   R(A) := Gbl[Kst(Bx)]    -- gone in 5.2 (-> GETTABUP _ENV)
OP_SETGLOBAL  A Bx   Gbl[Kst(Bx)] := R(A)    -- gone in 5.2 (-> SETTABUP _ENV)
OP_LOADNIL    A B    R(A) := ... := R(B):=nil-- range form (5.2 changed to A B count form)
OP_NEWTABLE / OP_SELF / arithmetic OP_ADD..OP_POW / OP_UNM / OP_NOT / OP_LEN / OP_CONCAT ...
```
Source: https://www.lua.org/source/5.1/lopcodes.h.html .

**5.2 opcode set** drops `GETGLOBAL`/`SETGLOBAL`, adds `GETTABUP`/`SETTABUP` (table access through
an upvalue, used for `_ENV`) and `LOADNIL` semantics change, and `TFORCALL`/`TFORLOOP` reshaping.
5.4's opcode set is different again (integer ops, `BANDK`/`SHRI`, etc.).

**Backend implication.** The unified VM's instruction decoder needs a per-version opcode table.
The cheapest viable plan: keep the internal IR/VM modern (5.4-style `_ENV`/closures) and have the
5.1 *loader/compiler* either (a) target a 5.1 opcode enum decoded by a 5.1-specific dispatch, or
(b) lower `OP_GETGLOBAL`/`SETGLOBAL` into the existing `_ENV`-upvalue table-access path at load
time. Option (b) maximizes code sharing (the bridge thesis) but must still expose 5.1-correct
`getfenv`/`setfenv` observable behavior. Structural-oracle parity (matching `luac` byte output)
is **not** required for the embedding product — we are past C-mirroring; only behavioral parity
against the reference binary matters.

## 6. Metamethods

| Metamethod | 5.1 | 5.2 | 5.4 | Critical note |
|---|---|---|---|---|
| `__index`/`__newindex` | ✔ | ✔ | ✔ | unchanged |
| `__add`…`__pow`,`__unm`,`__concat` | ✔ | ✔ | ✔ | arithmetic always float-dispatched in 5.1/5.2 |
| `__eq`,`__lt`,`__le` | ✔ | ✔ | ✔ | 5.4 dropped the `__lt`-derives-`__le` fallback (5.4 §2.4); 5.1/5.2 keep it |
| `__call` | ✔ | ✔ | ✔ | |
| `__len` | **userdata only** | tables **and** userdata | tables and userdata | **5.1 `len_event`: for a `table` returns primitive `#op` and never consults `__len`**; 5.2 `len_event` consults `metatable(op).__len` first (5.1 §2.8 vs 5.2 §2.4 pseudo-code) |
| `__pairs` | ✘ | ✔ | ✘ | added 5.2, **removed 5.4** (5.4 §8.3) |
| `__ipairs` | ✘ | ✔ | ✘ | added 5.2, removed 5.4 (deprecated 5.3) |
| `__gc` (tables) | **userdata only** | tables **and** userdata | tables and userdata | 5.2 added table finalizers; **must set `__gc` before/at metatable assignment** or the object is never finalized (5.2 §2.5.1, verbatim) |
| `__gc` (userdata) | ✔ | ✔ (order changed) | ✔ | 5.2 §8.3: userdata finalizers run in **reverse order of marking**, not creation order |
| `__metatable` | ✔ | ✔ | ✔ | hides/locks metatable |
| `__mode` (weak) | ✔ | ✔ (ephemeron) | ✔ | see §7 |
| `__name` | ✘ | ✘ | ✔ (5.3+) | tostring tag for userdata |
| integer-div/bitwise metas (`__idiv`,`__band`…) | ✘ | ✘ | ✔ (5.3+) | n/a for 5.1/5.2 |

The `__len`-on-tables divergence is the single most likely source of silent 5.1 test failures:
in 5.1, defining `__len` on a table metatable does **nothing** for the `#` operator; only
`newproxy()`-style userdata can intercept `#`. The backend's `#` dispatch must branch on version.

## 7. GC observable differences

- **Weak tables / ephemerons.** 5.2 §8.1 (verbatim): "Weak tables with weak keys now perform
  like ephemeron tables." In 5.1 a weak-key table does *not* have ephemeron semantics, so a value
  reachable only through its own weak key can survive a cycle in 5.1 but be collected in 5.2+.
  Observable via reachability tests in weak-table stress code.
- **Table finalizers.** 5.1 cannot finalize tables at all (no `__gc` on tables); the only way to
  observe collection is `__gc` on userdata (real or `newproxy`). 5.2+ finalizes tables.
- **Finalization order.** 5.2 §8.3: userdata `__gc` now runs in reverse order of *marking*, not
  of *creation*. Code that depends on `__gc` firing order will diverge between 5.1 and 5.2.
- **`collectgarbage` step/restart.** 5.2 §8.1: "Doing a step or a full collection in the garbage
  collector does not restart the collector if it has been stopped." So `collectgarbage('stop')`
  followed by `collectgarbage('step')` keeps the collector stopped in 5.2 but could effectively
  restart in 5.1.
- **`collectgarbage` options.** 5.1 has `collect/stop/restart/count/step/setpause/setstepmul`;
  `gcinfo()` is the deprecated 5.0 holdover. The incremental-GC tunables differ from 5.4's
  generational mode (5.4 added `"generational"`/`"incremental"` options + `"isrunning"`).
- These are **observable** through the official suites' GC stress tests and via
  `collectgarbage('count')` deltas — i.e. they are oracle-visible, not just internal.

## 8. Oracles — pinning, sources, reference binaries

**Pins:** Lua **5.1.5** and Lua **5.2.4**.

### Source tarballs (build the reference binaries)
- 5.1.5: https://www.lua.org/ftp/lua-5.1.5.tar.gz
- 5.2.4: https://www.lua.org/ftp/lua-5.2.4.tar.gz
- Browsable source (for spec cross-checks): https://www.lua.org/source/5.1/ and
  https://www.lua.org/source/5.2/
- Build (POSIX): `make linux` (or `make macosx`) → `src/lua` and `src/luac`. No external deps.
  Record the pinned commit/tarball sha + build command in the per-version `source.toml`, the same
  way the 5.4 reference is pinned. The reference `lua` binary is the **behavioral oracle**; `luac`
  is the **structural oracle** (but see §4 — structural parity is not a product requirement here).

### Test suites — the 5.1 suite is a different shape
The modern official test suite (`testes/`, the one used for 5.4 at ~75% pass) was reorganized
across versions. For 5.1/5.2 the matching suites are:
- 5.2.4 ships the modern-style `tests/` bundle (the `lua-5.2.x-tests` package on lua.org):
  https://www.lua.org/tests/ (the `lua-5.2.4-tests.tar.gz` style bundle). Shape is close to 5.4's
  `testes/` — `all.lua` driver plus per-feature files — so the existing harness wrapper adapts
  with minor edits.
- **5.1 has no official modern test bundle of the same shape.** The 5.1-era tests are older,
  smaller, and structured differently (the historical `test/` directory in the 5.1 tarball plus
  the separately distributed test scripts). Expect to **not** get a drop-in `all.lua` parity run;
  plan a curated 5.1 behavioral corpus (per-feature `.lua` files run through the
  diff-stdout-and-exit-code behavioral oracle against `lua-5.1.5`) rather than a single suite.
- Cross-version caution: many 5.4 `testes/` files use integer subtype, `goto`, `\u`, bitwise
  operators, `<const>` — they will **not** load under 5.1/5.2 and cannot be reused as-is. Build
  the 5.1/5.2 corpora from the version-matched suites, not by down-porting 5.4 tests.

## 9. Bridge thesis, restated

Port **5.2 first**: reuse the existing 5.4 `_ENV`/closure/globals machinery (5.2's globals are
the modern model) but swap the numeric core to float-only and adjust the stdlib surface
(`table.unpack`, `bit32`, `__pairs`/`__ipairs`, `__len`/`__gc` on tables, hex floats, goto, the
new escapes). That single backend proves the float-only core in isolation from the env model.
Then **5.1** is reached by *removing* from 5.2: drop `_ENV`/goto/`\x`/`\z`/hex-float, restore
function-environment globals + `setfenv`/`getfenv` + `OP_GETGLOBAL`/`SETGLOBAL`, restore the
legacy stdlib names (`loadstring`, global `unpack`, `table.getn/setn/maxn`, `module`/
`package.seeall`/`package.loaders`, `string.gfind`, `math.log10/atan2/pow`, one-arg `math.log`,
`newproxy`, `arg`), and flip `__len`/`__gc`/`__pairs` back to userdata-only / absent. The shared
seams between 5.2 and 5.1 are large (number core, most of stdlib, lexer minus a few tokens); the
genuine splits are globals dispatch, `__len`/`__gc` table eligibility, and the legacy opcode set.
