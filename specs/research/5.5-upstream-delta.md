# Lua 5.5 as a forward delta from Lua 5.4

Research spec for the WebLua multi-version embedding effort. All external claims
are cited. Primary sources are the upstream Lua 5.5 reference manual, the
`lua/lua` GitHub source pinned at tag `v5.5.0`, and the lua.org version page.

---

## FREEZE STATUS  (read this first)

**Lua 5.5.0 is a stable, frozen, officially released version. It is NOT a
moving "work" version, not an rc, not a beta.**

- Released **22 December 2025**. Source: lua.org versions page —
  "Lua 5.5.0 ... released on 22 Dec 2025" (https://www.lua.org/versions.html).
- This is the first 5.5 point release; it landed ~5.5 years after Lua 5.4
  (5.4.0 released 29 Jun 2020; latest 5.4 maintenance release is 5.4.8, 4 Jun
  2025) (https://www.lua.org/versions.html).
- Pin target: GitHub `lua/lua` tag **`v5.5.0`**. The tag resolves to commit
  `a5522f06d2679b8f18534fd6a9968f7eb539dc31` ("GC checks stack space before
  running finalizer", 2025-12-13) in the GitHub mirror
  (https://github.com/lua/lua/tree/v5.5.0). Version macros confirm the target:
  `LUA_VERSION_MAJOR_N 5`, `LUA_VERSION_MINOR_N 5`, `LUA_VERSION_RELEASE_N 0`
  (`lua.h`, https://github.com/lua/lua/blob/v5.5.0/lua.h).
- A 5.5.0 **beta** was announced on lua-l on the path to this release
  ("[ANN] Lua 5.5.0 (beta) now available",
  https://groups.google.com/g/lua-l/c/N1MMWqG4Ad0); the beta is now superseded
  by the final 5.5.0.

**Stability verdict for our purposes:** 5.5.0 is a *safe* target — the language
and bytecode are frozen for the 5.5.0 release the same way every prior Lua
point release is. Caveats that still bite us:
1. Lua provides **no stability guarantee across minor versions** and explicitly
   warns that precompiled chunks and the C ABI are never compatible across
   versions, and that all `LUA_COMPAT_*` options "will be removed in the future"
   (manual §8, https://www.lua.org/manual/5.5/manual.html#8). A future 5.5.x
   maintenance release can still patch behavior (5.4 went to 5.4.8).
2. The official **test suite (`testes/`) is not part of the released tarball**
   and has historically been distributed separately and informally (see Oracle
   section). Treat the GitHub `v5.5.0/testes` tree as the oracle but expect it
   to be lightly versioned.

Bottom line: 5.5 is release-grade and a defensible first multi-version backend
target. It is NOT more of a moving target than 5.4 was.

## 5.4 -> 5.5 delta table

| Area | 5.4 behavior | 5.5 behavior | Where it hits our code | Severity |
|---|---|---|---|---|
| `global` keyword | `global` is an ordinary identifier | `global` is a **reserved word** (`TK_GLOBAL`); new `global` statement declares globals | lexer (new token), parser (new statement), scope resolver | High |
| Global name resolution | every free name is implicitly `_ENV.name` | chunks start with implicit `global *`; once an explicit `global` decl appears in scope, undeclared free names are a **compile-time error** | scope resolver, codegen (new check opcode) | High |
| `for` control variable | numeric/generic loop var is a writable local | loop control var is **read-only (const)**; assigning to it is a compile error | parser (mark const), error path | Medium |
| Named vararg tables | `...` only, accessed via `select`/`table.pack` | `function f(a, ...t)` binds varargs to a named table `t`; new `OP_GETVARG` | parser (`parlist`), codegen, VM | Medium |
| `__call` chain depth | unspecified/long | a chain of `__call` metamethods is capped at **15 objects** | VM call path | Low |
| `nil` error object | error object can be `nil` | a `nil` error object is **replaced by a string message** | error/`pcall` path | Low |
| Bytecode version | `LUAC_VERSION == 0x54` (5*16+4) | `LUAC_VERSION == 0x55` (5*16+5 = 85) | `lundump`/`ldump` reader/writer | High (structural oracle) |
| Opcodes | 83 opcodes | **85 opcodes**: adds `OP_GETVARG`, `OP_ERRNNIL`; `OP_SHRI`/`OP_SHLI` order swapped | VM opcode enum, dispatch | High |
| Operand encoding | all `iABC` use fixed B(8)/C(8) | new **`ivABC`** mode: `vB(6)`/`vC(10)`; `OP_NEWTABLE`/`OP_SETLIST` now use vB/vC | instruction decode, codegen | High |
| `table.create` | absent | **new**: `table.create(nseq [, nrec])` preallocates | stdlib table | Low |
| `utf8.offset` | returns one value | also **returns the final position** of the character | stdlib utf8 | Low |
| GC tuning API | `collectgarbage("incremental"/"generational", ...)` set params | params set via new **`collectgarbage("param", ...)`**; C side: `LUA_GCPARAM` replaces `LUA_GCINC`/`LUA_GCGEN`; param set changed | GC tuning surface | Medium |
| Float `tostring` | shortest-ish | floats printed in decimal **with enough digits to round-trip** | number-to-string | Low |
| `lua_call` results cap | unspecified | max `nresults` is **250** (use `LUA_MULTRET` beyond that) | embedding API | Low |
| `lua_newstate` | 2 params | **3rd param**: string-hashing seed | embedding API | Medium (API) |
| Deprecated C API | — | `lua_resetthread`, `lua_setcstacklimit` deprecated; `lua_pushvfstring` now reports (not raises) errors; `lua_dump` calls writer one extra time | embedding API | Medium |
| New C API helpers | — | `luaL_openselectedlibs`, `luaL_makeseed`; external strings (memory not managed by Lua) | embedding API | Low |
| Memory | — | large arrays use ~60% less memory; major GC done incrementally | internal only (not behavioral) | n/a |

## 1. The global-declaration change (PRECISE)

This is the headline language change and the one that most affects our
lexer/parser/scope-resolver/codegen.

### 1a. `global` becomes a reserved word

`global` is now in the reserved-word table. In `llex.h` the `RESERVED` enum gains
`TK_GLOBAL` (between `TK_FUNCTION` and `TK_GOTO`), and `llex.c`'s reserved-word
array lists `... "function", "global", "goto", "if" ...`
(https://github.com/lua/lua/blob/v5.5.0/llex.c). Manual §8.1: *"The word
`global` is a reserved word. Do not use it as a regular name."*
(https://www.lua.org/manual/5.5/manual.html#8).

**Backward-compat hook:** `luaconf.h` defines `LUA_COMPAT_GLOBAL` *on by default*
("avoids `global` being a reserved word"). When set, the lexer un-reserves the
word: `llex.c` keeps a `glbn` ("global" name) `TString` and does
`ls->glbn->extra = 0; /* mark it as not reserved */`
(https://github.com/lua/lua/blob/v5.5.0/llex.c). So with the compat flag a
program may still use `global` as an identifier; the `global` *statement* is
recognized contextually. Like all `LUA_COMPAT_*`, this is slated for removal.

### 1b. Grammar (manual §3.3.7, Variable Declarations)

```
stat       ::= local  attnamelist ['=' explist]
stat       ::= global attnamelist ['=' explist]
attnamelist ::= [attrib] Name [attrib] {',' Name [attrib]}
attrib     ::= '<' Name '>'
stat       ::= global [attrib] '*'
```
(https://www.lua.org/manual/5.5/manual.html#3.3.7). Attributes are `<const>`
and `<close>`; **`<close>` is local-only** — `global X <close>` is a compile
error ("global variables cannot be to-be-closed"; see `getglobalattribute` in
`lparser.c`). A `global` declaration with initialization raises a **runtime**
error if the global already holds a non-nil value (manual §3.3.7).

### 1c. Name resolution: declared vs undeclared (manual §2.2)

The key model (manual §2.2, Scopes/Variables/Environments,
https://www.lua.org/manual/5.5/manual.html#2.2):

- **Every chunk starts with an implicit `global *`** declaration. While that
  preambular declaration is active, any free name is a normal global
  (`_ENV.name`), exactly like 5.4.
- An explicit `global` declaration **voids** the implicit `global *` for the
  rest of that scope. Inside such a scope, **every free name must be declared**
  (as local, upvalue, or in a `global` declaration); an undeclared free name is
  a **compile-time error**.
- `global *` (optionally `global<const> *`) re-enables "everything is a global
  by default" for the scope (and can make all such globals read-only).

Manual example (verbatim, §2.2):
```lua
X = 1           -- Ok, global by default
do
  global Y      -- voids the initial implicit declaration
  Y = 1         -- Ok, Y declared as global
  X = 1         -- ERROR, X not declared
end
X = 2           -- Ok, global by default again
```

### 1d. What it touches in our pipeline (mapped to upstream source)

- **Lexer (`lua-lex`)**: add `Global` token; honor a compat switch that demotes
  it to an identifier (mirrors `LUA_COMPAT_GLOBAL` / `glbn`).
- **Parser (`lua-parse`)**: new `globalstat` (`lparser.c:globalstat`,
  `globalnames`, `getglobalattribute`); a `global` decl carries a per-scope flag
  that switches the resolver into "must-declare" mode and tracks the declared
  global set + their attributes. Handles the `global *` collective form.
- **Scope resolver**: replace "all free names are globals" with the stateful
  model in §2.2. Upstream encodes this with `VGLOBAL` and a sentinel
  `var->u.info == -1` meaning "preambular `global *` is active"
  (`lparser.c` comments around `buildglobal`/`singlevaraux`). Read/write of a
  `<const>` global must error at compile time.
- **Codegen (`lua-code`)**: undeclared-global *checks* are emitted as a new
  opcode, **`OP_ERRNNIL`** — `A Bx  raise error if R[A] ~= nil` where `K[Bx-1]`
  is the global name; if `Bx == 0` the name didn't fit and is omitted from the
  message (`lopcodes.h`, https://github.com/lua/lua/blob/v5.5.0/lopcodes.h).
  `lparser.c:checkglobal`/`luaK_codecheckglobal` emit it.

## 2. Numeric for-loop semantic changes

- **Control variable is read-only.** Manual §8.1: *"The control variable in
  `for` loops is read only. If you need to change it, declare a local variable
  with the same name in the loop body."* (https://www.lua.org/manual/5.5/manual.html#8).
  The variable is a fresh `<const>` local scoped to the loop body. Rationale per
  lua-l: 5.4 did an implicit `local x = x` copy on every loop to guard against
  mutation; 5.5 only does that copy when you ask, saving overhead
  (https://groups.google.com/g/lua-l/c/SlAG5QfpTac).
- Enforcement: `lparser.c:check_readonly` raises a compile error on assignment to
  the loop var (it is marked `ro`).
- **Integer overflow semantics are unchanged from 5.4**: an integer numeric-for
  never wraps; the loop terminates on overflow of the control variable. (Same
  wording in both 5.4 and 5.5 manuals, §3.3.5;
  https://www.lua.org/manual/5.4/manual.html#3.3.5,
  https://www.lua.org/manual/5.5/manual.html#3.3.5.) Float loops use a precomputed
  iteration count (`floatforloop` in `lvm.c`). So our existing for-loop count
  logic carries over; only the read-only marking is new.

## 3. Other syntax additions / removals

- **Named vararg tables.** `parlist` now accepts `function f(a, ...t)`: the `...`
  may be followed by a name, binding the varargs to a table-like parameter
  (`lparser.c:parlist`, kind `RDKVAVAR`,
  https://github.com/lua/lua/blob/v5.5.0/lparser.c). Indexing the named vararg
  parameter compiles to `OP_GETVARG` (`R[A] := R[B][R[C]]`, R[B] is the vararg
  parameter). Plain `...` (anonymous) still works. The vararg table is "virtual"
  / read-only unless materialized (`check_readonly` calls `needvatab`).
- **More levels for constructors** — table constructors allow deeper nesting
  (readme.html "Changes"). No grammar change, a limit bump.
- **No removals** of language constructs beyond the `global` reservation.

## 4. Bytecode / opcode changes from 5.4

These matter because our **structural oracle** is `luac`-style bytecode bytes.

- **Header / version:** `LUAC_VERSION = LUA_VERSION_MAJOR_N*16 + LUA_VERSION_MINOR_N
  = 0x55 (85)`, vs 5.4's `0x54`. `LUAC_FORMAT 0`, `LUAC_DATA "\x19\x93\r\n\x1a\n"`
  unchanged (`lundump.h`, https://github.com/lua/lua/blob/v5.5.0/lundump.h).
  Precompiled chunks are NOT cross-version compatible (manual §8).
- **Opcode count 83 -> 85.** Two new opcodes (`lopcodes.h`):
  - `OP_GETVARG` — `A B C  R[A] := R[B][R[C]]`, R[B] is the vararg parameter
    (powers named vararg indexing).
  - `OP_ERRNNIL` — `A Bx  raise error if R[A] ~= nil (K[Bx-1] is global name)`
    (powers the undeclared/declared-global compile-time-emitted check; `Bx==0`
    means the name didn't fit and is dropped from the message).
- **Opcode reordering:** `OP_SHRI` and `OP_SHLI` swapped order relative to 5.4.
  Any table keyed by raw opcode index must be regenerated for 5.5.
- **New instruction format `ivABC`.** `OpMode` gains `ivABC`. Layout:
  `iABC` is `C(8) B(8) k A(8) Op(7)`; `ivABC` is `vC(10) vB(6) k A(8) Op(7)`
  (so `SIZE_vC=10`, `SIZE_vB=6`, `MAXARG_vC=1023`, `MAXARG_vB=63`). `OP_NEWTABLE`
  and `OP_SETLIST` switched to `ivABC` to support the more-compact-array work:
  - `OP_NEWTABLE A vB vC k` — vB is `log2(hash size)+1` (or 0); array size is `vC`,
    or `EXTRAARG _ vC` if `k`.
  - `OP_SETLIST A vB vC k` — `R[A][vC+i] := R[A+i], 1 <= i <= vB`.
  (`lopcodes.h`, https://github.com/lua/lua/blob/v5.5.0/lopcodes.h.) Our
  instruction decoder and codegen must add this mode and route NEWTABLE/SETLIST
  through it.
- **dump/undump:** `lua_dump` now calls the writer one extra time (end-of-dump
  signal) and changed stack handling; dump/undump reuse all strings; binary
  chunks loaded in memory can reuse original memory for some internal structures
  (manual §8.3, readme.html). Relevant only if we implement faithful
  serialize/deserialize.

## 5. Stdlib additions / removals / changes

- **`table.create(nseq [, nrec])`** — NEW. Preallocates a table with `nseq`
  array slots and `nrec` hash slots (`ltablib.c`,
  https://github.com/lua/lua/blob/v5.5.0/ltablib.c; readme.html lib entry).
- **`utf8.offset`** — now ALSO returns the final byte position of the character,
  in addition to its start (readme.html "Changes"; `lutf8lib.c:byteoffset`).
- **`collectgarbage`** — the `"incremental"` and `"generational"` options NO
  LONGER take tuning parameters; tuning moves to a new **`collectgarbage("param",
  ...)`** option, and several GC parameters themselves changed (manual §8.2,
  https://www.lua.org/manual/5.5/manual.html#8).
- **Float formatting** — `tostring`/`string.format %g`-style float output now
  prints enough decimal digits to round-trip exactly (readme.html "Changes").
- **No library functions were removed.** The deprecated `math` aliases (`atan2`,
  `cosh`, `sinh`, `tanh`, `pow`, `ldexp`, `frexp`, `log10`) still exist, gated by
  `LUA_COMPAT_MATHLIB` (off by default in `luaconf.h`) — same situation as 5.4.
- New C-side library plumbing (affects embedding more than Lua scripts):
  `luaL_openselectedlibs(L, load, preload)` and `luaL_makeseed(L)`
  (`linit.c`, `lauxlib.h`, https://github.com/lua/lua/blob/v5.5.0/lauxlib.h).

## 6. Metamethod / GC / error-message changes

- **Metamethods:** the `TM_*` enum is **identical** between 5.4.7 and 5.5.0 — no
  metamethods added or removed (diff of `ltm.h`). The only metamethod-related
  change is a behavioral cap: a **chain of `__call` metamethods is limited to 15
  objects** (manual §8.1).
- **GC:**
  - Major (full) collections are now done **incrementally** (readme.html).
  - Tuning API changed (see §5 / §8.2): `collectgarbage("param", ...)` replaces
    parameter passing through `"incremental"`/`"generational"`; C side
    `LUA_GCPARAM` replaces `LUA_GCINC`/`LUA_GCGEN` (manual §8.2, §8.3).
  - Generational mode and a new GC parameter set; specific numeric params
    changed (manual §8). Internal-only unless we mirror tuning knobs.
- **Error messages / error object:**
  - A `nil` error object is now **replaced by a string message** (manual §8.1) —
    so `error(nil)` propagated through `pcall` yields a string, not `nil`. This
    is observable and our `pcall`/`error` path must match.
  - `lua_pushvfstring` now **reports** errors instead of raising them
    (manual §8.3) — embedding-API only.

## ORACLE for Lua 5.5

### Source + build (reference binary)
- **Source tarball:** `https://www.lua.org/ftp/lua-5.5.0.tar.gz` (canonical,
  per lua.org/versions.html — 5.5.0 released 22 Dec 2025).
- **Git pin:** GitHub `lua/lua` tag **`v5.5.0`**
  (https://github.com/lua/lua/tree/v5.5.0). Note the GitHub mirror lays sources
  at the **repo root** (e.g. `./lua.h`, `./lopcodes.h`), not under `src/`, unlike
  the tarball which uses `src/`.
- **Build (matches our existing 5.4 reference flow):**
  ```sh
  # tarball layout
  make all            # or: make macosx / make linux / make posix
  # GitHub-mirror layout (sources at root)
  make -f makefile macosx   # MYCFLAGS uses -std=c99; LUA_USE_* per platform
  ```
  The platform target sets `LUA_USE_MACOSX`/`LUA_USE_LINUX`/`LUA_USE_POSIX`.
  Compat flags `LUA_COMPAT_GLOBAL` (on) and `LUA_COMPAT_MATHLIB` (off) are
  defaulted in `luaconf.h`; for a strict oracle, **build a second binary with
  all `LUA_COMPAT_*` off** to catch the `global`-as-keyword behavior we will
  ultimately implement.
- Resulting binaries: `lua` (interpreter) and `luac` (compiler/dumper, our
  structural-oracle byte source).

### Freeze-status caveats for the oracle
- 5.5.0 is frozen, but a future 5.5.x maintenance release may patch behavior
  (5.4 reached 5.4.8). Pin tests to the exact tag you build.
- Precompiled-chunk bytes and the C ABI are explicitly non-portable across
  versions (manual §8) — keep the 5.5 structural-oracle bytes separate from 5.4.
- The default-on `LUA_COMPAT_GLOBAL` means the stock reference binary will *not*
  reject `global` as an identifier; decide per-test which build (compat on vs
  off) is the oracle for a given behavior.

### Official test suite (behavioral oracle)
- Distributed separately from the release tarball (historically posted to lua-l
  / available as `lua-5.5.0-tests.tar.gz`-style bundles). The GitHub `v5.5.0`
  tree includes a `testes/` directory we can use directly.
- Entry point: `testes/all.lua` (run from inside `testes/` with the built `lua`;
  many files require the C test libs under `testes/libs/` and the
  `-DLUA_USER_H='"ltests.h"'` debug build for `api.lua`/`db.lua`).
- **Test file list (`testes/*.lua`, 34 files):**
  `all.lua, api.lua, attrib.lua, big.lua, bitwise.lua, bwcoercion.lua, calls.lua,
  closure.lua, code.lua, constructs.lua, coroutine.lua, cstack.lua, db.lua,
  errors.lua, events.lua, files.lua, gc.lua, gengc.lua, goto.lua, heavy.lua,
  literals.lua, locals.lua, main.lua, math.lua, memerr.lua, nextvar.lua, pm.lua,
  sort.lua, strings.lua, tpack.lua, tracegc.lua, utf8.lua, vararg.lua,
  verybig.lua`
  plus C helper libs in `testes/libs/` (`lib1.c, lib2.c, lib11.c, lib21.c,
  lib22.c`) and a `makefile`.
- Files most relevant to the 5.5 delta: `locals.lua` (global decls, `<const>`),
  `attrib.lua` (attributes), `vararg.lua` (named vararg tables), `code.lua`
  (bytecode/opcode shape), `nextvar.lua`/`gc.lua`/`gengc.lua` (table + GC),
  `errors.lua` (nil error object, messages), `math.lua` (float round-trip),
  `utf8.lua` (`utf8.offset` second return).

## Top 5 risks (for porting 5.5 as a backend)

1. **Bytecode-shape divergence is broad, not local.** New `ivABC` operand mode +
   reordered `SHRI`/`SHLI` + two new opcodes (`GETVARG`, `ERRNNIL`) + version
   byte `0x55` mean the structural oracle and any opcode-indexed tables must be
   regenerated wholesale; a "diff against 5.4" assumption will silently corrupt
   decode.
2. **Global-decl scope model is genuinely stateful.** "Implicit `global *`,
   voided by an explicit `global` decl, re-enabled by `global *`" is a per-scope
   mode flag with compile-time errors — not a cosmetic keyword. Easy to get the
   nesting/voiding rules subtly wrong; the `LUA_COMPAT_GLOBAL` default-on adds a
   second behavior axis to test.
3. **Shared-code-across-versions tension.** `global` keyword, read-only for-vars,
   and named vararg tables are *parser/codegen* divergences from 5.4, exactly the
   layer the embedding-API vision wants to share. These force a real
   version-split at lex/parse/codegen, not just a stdlib shim.
4. **GC tuning API contract changed** (`collectgarbage("param", ...)`, `LUA_GCPARAM`)
   and full GC now incremental — if we expose any GC knobs through the unified
   embedding API, the 5.4 vs 5.5 surfaces differ and tests like `gc.lua`/`gengc.lua`
   will exercise the new params.
5. **Oracle logistics.** The test suite ships separately from the release and is
   only lightly versioned; the GitHub mirror's root-level source layout differs
   from the tarball's `src/` layout; and the default-on compat flag means the
   stock binary won't reject `global` as an identifier — all of which can produce
   false oracle agreements/disagreements unless the reference build and test
   bundle are pinned together explicitly.
